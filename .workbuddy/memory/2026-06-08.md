# 2026-06-08 安全审查与修复

## 审查发现的问题

### 🔴 高危漏洞（已修复）

| # | 问题 | 风险 | 修复方式 |
|---|------|------|----------|
| 1 | **支付接口无验证** | 任何人直接 POST `/api/confirm-payment` 即可无限增加付费次数 | 添加支付 token 机制：弹窗前获取 token → 确认时验证 token 签名和有效期 |
| 2 | **配置接口无身份验证** | 任何人可修改系统 API Key、模型、价格 | 添加 `X-Admin-Password` Header 验证，未配置时完全禁用修改 |
| 3 | **User-ID 可被伪造** | 修改 `X-User-Id` Header 即可冒充他人消耗次数 | 添加 HMAC-SHA256 签名机制，`X-User-Sig` 与 `X-User-Id` 绑定验证 |
| 4 | **CORS 完全开放** | `Access-Control-Allow-Origin: *` 允许任何网站跨域调用 API | 限制为只允许同源/localhost，外部来源不返回 CORS 头 |

### 🟡 中危问题（已修复）

| # | 问题 | 修复方式 |
|---|------|----------|
| 5 | **错误信息可能泄露 API 端点** | AI 调用错误时清理错误信息，不返回原始 API 响应 |
| 6 | **缺少频率限制** | 添加内存级 rate limiter（生成 5次/分钟，支付 3次/分钟） |

### ✅ 原本就安全的设计

- 系统 API Key 完全在后端处理，前端无任何接触
- config.json / users.json 无法通过 HTTP 直接访问
- 前端代码中无硬编码的系统 API Key
- 系统 API Key 不在任何 API 响应中返回
- 静态文件服务仅开放 public/ 目录

## 新增安全模块

| 文件 | 功能 |
|------|------|
| `server/utils/securityConfig.js` | 安全常量：ADMIN_PASSWORD、USER_ID_SECRET、PAYMENT_SECRET（从环境变量读取） |
| `server/utils/security.js` | User-ID 签名/验证、支付 token 生成/验证、频率限制、管理员验证中间件 |

## 修复后的 API 行为变化

### /api/status
- **请求头**: `X-User-Id` + `X-User-Sig`（可选，首次无 sig）
- **响应**: 返回 `userSig`，前端保存后后续请求必须带上
- **不再返回**: `userId`

### /api/generate
- **请求头**: `X-User-Id` + `X-User-Sig`（必须）
- **新增**: 频率限制（5次/分钟）
- **错误信息**: 清理后返回，不泄露内部 API 细节

### /api/confirm-payment
- **流程**: 先 GET `/api/confirm-payment/token` 获取 token → POST 时带上 token
- **Token 机制**: 包含时间戳 + HMAC 签名，5分钟有效期
- **频率限制**: 3次/分钟

### POST /api/config
- **需要**: `X-Admin-Password` Header
- **未配置时**: 返回 `admin_not_configured`，完全禁止修改

### CORS
- **外部来源**: 不返回 `Access-Control-Allow-Origin`，浏览器会阻止跨域请求
- **localhost/同源**: 正常允许

## 测试验证结果

- ✅ 伪造 userId + 错误 sig → 服务端生成新用户，无法消费他人次数
- ✅ 无 payment token 调用确认付款 → 返回 `invalid_token`
- ✅ 无管理员密码调用 config POST → 返回 `admin_not_configured`
- ✅ 外部 Origin 请求 API → 无 CORS 头，浏览器会阻止
- ✅ Config GET → 只返回非敏感字段，无实际 API Key
- ✅ 系统 API Key 不在任何响应中出现
